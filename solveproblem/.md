+# AI 工具调用异常、监控 NPE 与 SSE 二次异常修复记录
+
+## 问题现象
+
+在网站生成过程中，后端日志出现了以下几类异常：
+
+1. **工具调用参数解析失败**
+   - `ToolExecutionRequestUtil.argumentsAsMap`
+   - `JsonParseException: Unexpected character ('p')`
+
+2. **监控监听器空指针**
+   - `AiModelMonitorListener.onResponse`
+   - `AiModelMonitorListener.onError`
+   - `MonitorContext` 为 `null`
+
+3. **SSE 错误处理二次异常**
+   - `GlobalExceptionHandler`
+   - `getOutputStream() has already been called for this response`
+   - `No converter for [class java.util.LinkedHashMap] with preset Content-Type 'text/event-stream'`
+
+## 根因分析
+
+### 1. 工具调用参数 JSON 不稳定
+
+Vue 项目生成使用推理流式模型 + 工具调用。模型返回的工具参数 JSON 在部分情况下格式不规范，导致：
+
+- LangChain4j 在 `argumentsAsMap` 反序列化时报错
+- 工具调用链中断
+- 最终网站生成失败
+
+### 2. 监控上下文依赖 ThreadLocal，但流式回调跨线程
+
+`AiModelMonitorListener` 中依赖 `MonitorContextHolder.getContext()` 取上下文，但流式模型的请求、响应、错误回调不一定发生在同一个线程中，导致：
+
+- `onRequest` 可能拿不到上下文
+- `onResponse` / `onError` 可能拿不到上下文
+- 进而触发 NPE
+
+### 3. SSE 异常返回链路不够安全
+
+当流式响应已经开始写入或已经使用 `OutputStream` 后，`GlobalExceptionHandler` 仍尝试通过 `getWriter()` 写 SSE 错误消息，导致：
+
+- 原始业务异常之外又抛一次异常
+- 日志被污染
+- 前端更难判断真实失败原因
+
+## 修复方案
+
+### 1. 收紧推理模型的工具调用配置
+
+修改文件：
+
+- `src/main/java/com/yupi/yuaicodemother/config/ReasoningStreamingChatModelConfig.java`
+
+处理方式：
+
+- 开启 `strictJsonSchema(true)`
+- 开启 `strictTools(true)`
+- 关闭 `parallelToolCalls(false)`
+
+目的：
+
+- 提高工具参数 JSON 的规范性
+- 降低 `argumentsAsMap` 解析失败概率
+- 避免并行工具调用导致的混乱和写文件顺序问题
+
+### 2. 监听器在上下文缺失时安全降级
+
+修改文件：
+
+- `src/main/java/com/yupi/yuaicodemother/monitor/AiModelMonitorListener.java`
+
+处理方式：
+
+- `onRequest` 中取不到 `MonitorContext` 时直接跳过监控采集
+- `onResponse` / `onError` 优先从 `attributes` 取上下文
+- 若 `attributes` 中没有，再回退到 `ThreadLocal`
+- 若仍为空，则直接返回，不再抛异常
+- `recordResponseTime` 中补充 `startTime == null` 判断
+
+目的：
+
+- 保证监控是“可选能力”，而不是让主流程失败
+- 避免流式跨线程带来的 NPE
+
+### 3. SSE 异常处理增加防御逻辑
+
+修改文件：
+
+- `src/main/java/com/yupi/yuaicodemother/exception/GlobalExceptionHandler.java`
+
+处理方式：
+
+- `response == null` 时直接返回
+- `response.isCommitted()` 时直接认为已处理
+- 捕获 `IllegalStateException`
+- 在无法继续写 SSE 时不再向外抛二次异常
+
+目的：
+
+- 避免 `getWriter()` / `getOutputStream()` 冲突
+- 减少异常链路上的噪音日志
+- 保留原始错误作为主要排查线索
+
+## 回归测试
+
+新增测试文件：
+
+- `src/test/java/com/yupi/yuaicodemother/monitor/AiModelMonitorListenerTest.java`
+- `src/test/java/com/yupi/yuaicodemother/exception/GlobalExceptionHandlerTest.java`
+- `src/test/java/com/yupi/yuaicodemother/config/ReasoningStreamingChatModelConfigTest.java`
+
+验证命令：
+
+```bash
+mvn "-Dtest=AiModelMonitorListenerTest,GlobalExceptionHandlerTest,ReasoningStreamingChatModelConfigTest" test
+```
+
+验证结果：
+
+- 监听器空上下文时不再 NPE
+- SSE 响应流类型冲突时不再二次抛错
+- 推理模型工具调用配置已按预期开启严格模式
+
+## 结论
+
+这次问题的**直接根因**是工具调用参数 JSON 解析失败；  
+监控 NPE 和 SSE 二次异常是错误路径上的派生问题。
+
+修复后，主流程在以下方面更稳定：
+
+- 工具调用参数更规范
+- 监控缺失不再影响主业务
+- SSE 异常处理更安全
